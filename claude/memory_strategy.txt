# Memory Management Strategy for Claude

## Hierarchy of Information Access

### 1. CLAUDE.md (Always First)
- **When**: Start of every conversation
- **What**: Critical rules, quick reference
- **Why**: Persistent, project-specific guidelines

### 2. Recent Logs Check
- **When**: Before any development task
- **What**: Last 10-20 entries in `/claude/logs.md`
- **Why**: Understand recent changes and context

### 3. Content.md Reference
- **When**: Before creating/modifying files
- **What**: Current project structure
- **Why**: Prevent duplicates, find existing code

### 4. Working Memory Usage
- **What to track**:
  - Current task progress
  - Recent file modifications
  - Discovered issues/blockers
  - Test results
- **What NOT to store**:
  - Large code blocks (use file references)
  - Repetitive information from CLAUDE.md
  - Full file contents (use Read when needed)

## Optimal Workflow

1. **Session Start**:
   ```
   Read CLAUDE.md → Check recent logs → Review current todos
   ```

2. **During Development**:
   ```
   Check content.md → Make changes → Log immediately → Update todos
   ```

3. **Context Switching**:
   ```
   Save state in todos → Document in logs → Quick summary in response
   ```

## Key Principles

- **CLAUDE.md** = Rules & quick reference (static)
- **Memory** = Current task state (dynamic)
- **Logs** = History & audit trail (append-only)
- **Content** = Project map (updated on structure changes)

## Red Flags to Avoid

❌ Forgetting to log file changes
❌ Creating files without checking content.md
❌ Storing full file contents in memory
❌ Not updating todos regularly
❌ Ignoring CLAUDE.md rules
